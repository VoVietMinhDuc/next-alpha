# OptiBot Mini-Clone

Support chatbot: ingests OptiSigns articles into clean Markdown, serves them via a
**Gemini File Search Store**. A daily job re-scrapes and uploads only what changed.

**Stack:** Python · Gemini File Search API (`google-genai`) · Docker · DigitalOcean droplet (cron).

📊 **Live daily-job logs:** http://178.128.98.214:8888/logs/

## Architecture

```
main.py               Daily job orchestrator (scrape -> delta -> upload), runs once & exits 0
src/scraper.py        Part 1: Zendesk API -> clean Markdown (<slug>-<id>.md)
src/uploader.py       Part 2: create/reuse Gemini File Search Store, upload via API
src/ask.py            Part 2: query the assistant (OptiBot prompt), grounded + cited
src/delta.py          Hash-based add/update/skip detection (state read from the store itself)
src/config.py         Env-based config (no hard-coded keys)
src/inspect_store.py  Dev helper: list store documents / run a grounded query
data/articles/        Generated .md files
logs/                 Per-run JSON artefacts (counts + store stats)
```

## Setup

```bash
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.sample .env        # then fill in API_KEY
```

> **⚠️ Reuse the store.** Leave `FILE_SEARCH_STORE_NAME` **blank on the first run**;
> the uploader creates a store and logs its name — **copy that into `.env`** so later
> runs reuse it instead of re-embedding from scratch (it's also what the delta diffs against).

## Run locally

```bash
python -m src.scraper      # Part 1 only: scrape -> data/articles/*.md
python -m src.uploader     # Part 2 only: upload to the File Search Store
python -m src.ask "How do I add a YouTube video?"   # Part 2: ask the assistant (cited answer)
python main.py             # Part 3: full daily job (scrape + delta + upload)
```

## Run with Docker

```bash
docker build -t optibot .

# Fill .env first (cp .env.sample .env). Secrets are passed at runtime, never
# baked into the image. Runs once and exits 0. Works as-is on PowerShell/bash/Linux/mac.
docker run --rm --env-file .env optibot

# ...or pass secrets inline instead of a file:
docker run --rm -e API_KEY=... -e FILE_SEARCH_STORE_NAME=... optibot
```

To keep each run's JSON artefact on the host, add a volume mount:
`-v "$(pwd)/logs:/app/logs"` (bash) or `-v "${PWD}/logs:/app/logs"` (PowerShell).

## Chunking strategy

- **500 tokens/chunk, 100 overlap** (`white_space_config`; Gemini caps a chunk at 512).
- Articles are short, so chunks stay **self-contained** (a step-by-step list won't split mid-way).
- Logs report the **exact** file count + the store's **API-reported** doc count / byte size; chunk count is an **estimate** (the API doesn't expose per-doc chunk counts).

## Delta & sync strategy

The daily job uploads **only what changed**: `delta.diff` hashes each article
(SHA-256) and classifies it as added / updated / skipped.

- **Stateless, idempotent** — each doc carries its content hash in `custom_metadata`; each run rebuilds the `{slug: hash}` map from the store (no state file), upserts by slug so re-runs never duplicate, and self-heals if a doc is deleted or an index fails.
- **Limitation:** removed articles aren't pruned (the brief's contract is add/update/skip).

## Daily job & logs

Dockerized one-shot, scheduled on a **DigitalOcean droplet** via
`crontab` (`0 0 * * *`, daily 00:00 UTC). Each run writes a JSON artefact with the
add/update/skip counts.
